# LongCat-Flash-Chat 560B至1.12万亿参数模型扩增方案与性能保持策略研究报告

在人工智能领域，模型规模的持续扩展被视为通往通用人工智能（AGI）的核心路径之一。基于美团 LongCat-Flash-Chat 560B 模型的现有架构，将其参数量翻倍至约 1.12 万亿（1.12T）不仅是简单的参数堆叠，更是一场涉及架构演进、通信优化、初始化策略以及大规模分布式训练稳定性的复杂系统工程。本报告旨在为这一扩增任务提供深度调研与全栈式技术方案，重点在于如何在实现参数规模翻倍的同时，通过稀疏激活机制、结构化上循环（Upcycling）和精细化愈合（Healing）训练，最大限度地保留甚至增强原模型的知识表征能力与推理性能 。

## 1. LongCat-Flash-Chat 560B 基座架构深度解析

在制定扩增方案之前，必须对 LongCat-Flash-Chat 560B（以下简称 LFC-560B）的现有架构进行微观层面的剖析。LFC-560B 是一款基于混合专家（Mixture-of-Experts, MoE）架构的大规模语言模型，其设计核心在于计算效率与智能涌现的平衡 。

### 1.1 稀疏专家配置与路由机制

LFC-560B 拥有 5600 亿总参数，但在推理过程中，每 token 激活的参数量仅为 186 亿至 313 亿，平均约为 270 亿 。这种高度稀疏的特性由其专家层配置决定：模型包含 512 个路由专家（Routed Experts），在每一层中通过 Top-12 路由算法选择最相关的专家参与计算 。此外，模型引入了 256 个“零计算专家”（Zero-computation Experts），这些专家作为占位符不参与实际矩阵运算，旨在通过动态计算预算分配，为不重要的 token 节省计算资源，从而在维持高吞吐量的同时确保关键 token 获得充足的计算权重 。

### 1.2 关键架构组件：MLA 与 ScMoE

LFC-560B 的另一个技术制高点是多头潜在注意力（Multi-head Latent Attention, MLA）。相比于传统的 KV 缓存机制，MLA 通过将键（Key）和值（Value）压缩到低维潜在空间，显著降低了长文本推理时的内存占用，其 KV 缓存压缩率高达 93.3% 。这种设计为模型支持 128k 甚至更长的上下文窗口提供了硬件基础 。

在专家并行通信方面，LFC-560B 采用了快捷连接 MoE（Shortcut-connected MoE, ScMoE）架构 。ScMoE 的核心创新在于解耦了传统 MoE 中通信与计算的串行依赖，通过引入前一层表示的快捷连接，使得 All-to-All 通信能够与当前层的计算实现 70% 至 100% 的重叠 。这一设计在大规模集群（如 H800/H100）上极大地缓解了网络带宽瓶颈，使得 560B 规模的模型仍能维持超过 100 TPS 的生成速度 。

## 2. 参数量翻倍的理论框架：混合专家上循环

针对 robin 提出的将 560B 模型参数翻倍至 1.12T 的需求，最有效的路径是执行“专家上循环”（Expert Upcycling）。与从零开始训练相比，上循环利用已训练模型的权重进行热启动，能够节省约 30%-40% 的计算资源，同时有效继承原模型的知识库 。

### 2.1 专家数量扩展逻辑

扩增至 1.12T 的最直接方法是将路由专家的数量从 512 个翻倍至 1024 个 。这一过程遵循算子 $U_m$ 的定义，其中 $m=2$ 表示扩增倍数。在保持 Top-12 路由不变的前提下，总参数量增加了一倍，但每 token 的激活计算量（FLOPs）基本保持恒定，从而在不显著增加单步推理延迟的情况下，极大提升了模型的“知识容量”和条件计算的精细度 。

### 2.2 路由器的平滑扩展

当专家数量增加时，路由权重矩阵 $W_g$ 必须相应扩展。扩增方案要求将原有的路由权重复制到新增加的专家槽位中。为了避免对称性导致的专家塌陷，必须在复制后的路由器权重中引入微小的偏置噪声 $\epsilon \sim \mathcal{U}(-\delta, \delta)$ 。这种随机性是打破专家间对称性的关键，使得在后续的持续预训练（CPT）中，原本完全相同的“双胞胎”专家能够逐渐演化并负责处理不同的语义模式 。

### 2.3 效用驱动的非均匀复制策略

尽管均匀复制是工业界最常用的基准方案，但更先进的调研表明，基于“效用”（Utility）的非均匀复制能够带来更优的收敛速度 。通过计算每个专家在验证集上的平方梯度范数（Squared Gradient Norm）或权重-梯度显著性（Weight-Gradient Saliency），可以识别出哪些专家承载了更多的知识负载 ：

$$u_{SAL}(e) = \|w_e\|_2 \cdot \|g_e\|_2$$

对于高显著性（High-utility）的专家，可以分配 3 个或更多的副本，而对于低显著性专家则仅保留 1 个副本或不进行复制 。在 1.12T 模型的扩增中，这种策略能确保新增的参数空间被分配到最需要复杂逻辑处理的领域，如数学推理和代码合成。

## 3. 结构化深度与宽度扩展方案

除了单纯增加专家数量，结构性扩展（Structural Scaling）也是达到 1.12T 目标的重要手段，尤其是在希望增强模型逻辑推理深度时 。

### 3.1 深度扩容：DUS 算法的应用

深度上缩放（Depth Up-Scaling, DUS）通过在现有的 Transformer 块之间插入重复的层来增加模型深度 。对于 LFC-560B 的 28/56 层结构，可以通过以下方式将其扩展：

1. **块复制与拼接**：将原模型的中间层进行复制，并利用恒等映射或平滑插值技术将其嵌入 。
2. **知识保留**：这种方法不改变矩阵维度，因此对现有推理引擎极其友好。 然而，深度的单纯增加会增加推理的串行延迟 。因此，在 1.12T 方案中，建议采用“专家扩增为主，深度微调为辅”的策略，以维持 LFC 系列的“Flash”推理特性。

### 3.2 宽度扩展：隐藏层维度增强

通过 Net2Net 或 HyperCloning 技术增加 $d_{model}$（隐藏层维度）可以显著提升模型在每一层的信息承载量 。将 $d_{model}$ 从 6,144 扩展至更高的维度（如 8,192），需要对权重矩阵进行神经元切分和复制 。为了保持函数等价性（Function-preserving），必须对扩增后的投影矩阵进行缩放处理，例如将查询（Query）权重缩放 $\sqrt{d_{old} / d_{new}}$，以确保注意力分布在初始化阶段不发生偏移 。

### 扩展策略对比分析表

| **扩展维度**   | **实现方法**           | **优势**               | **挑战**           | **来源** |
| -------------- | ---------------------- | ---------------------- | ------------------ | -------- |
| **专家扩增**   | 专家倍增 ($U_m$)       | 推理成本固定，容量激增 | 通信开销增加       |          |
| **深度扩增**   | DUS (深度上缩放)       | 逻辑推理能力增强       | 串行延迟增加       |          |
| **宽度扩容**   | Net2Net / HyperCloning | 单层表征能力提升       | 硬件内核兼容性挑战 |          |
| **零计算扩充** | Null Expert 翻倍       | 极端稀疏化，节省算力   | 路由不稳定性       |          |



## 4. LongCat 系列特有的效率扩增技术

由于 LFC-560B 拥有独特的 ScMoE 和零计算专家机制，在扩展至 1.12T 时，需要针对这些特性进行专项设计，以防性能下降 。

### 4.1 零计算专家的弹性扩展

LFC-560B 现有的 256 个零计算专家对于维持平均 27B 的激活规模至关重要 。在 1.12T 模型中，建议将零计算专家增加至 512 个。这一设计的深层意义在于：随着总专家数增加到 1024 个，模型对 token 的分类会更加细致，更多的 token 可以被路由到“跳过”槽位中，从而在大参数量下依然维持较低的硬件功耗和每 token 成本 。

### 4.2 ScMoE 的通信拓扑升级

在 1.12T 规模下，All-to-All 通信的节点规模将翻倍。ScMoE 必须配合更精细的专家并行（Expert Parallelism, EP）策略。研究显示，利用 Megatron-Core 的并行折叠（Parallel Folding）技术，可以将 EP 与数据并行解耦，使得注意力层和 MoE 层各自运行在最优的并行配置下 。此外，扩增后的 ScMoE 需要重新校准自适应重叠策略，确保在 H800 的 400 GB/s NVLink 带宽下，通信开销不占据主导地位 。

### 4.3 N-gram 嵌入表的集成可能性

借鉴 LongCat-Flash-Lite (68.5B) 的成功经验，该模型通过将 300 亿参数分配给 N-gram 嵌入表，实现了极强的短语记忆与快速推理能力 。在 1.12T 扩增计划中，如果不希望将全部 560B 的增量参数分配给专家层，可以考虑将约 100B-200B 参数用于构建更大规模的多标记 N-gram 查找表。这不仅能通过硬件缓存直接提升常见模式的处理速度，还能降低深层 Transformer 的计算负载 。

## 5. “愈合”训练（Healing）：性能保持的核心策略

参数翻倍后，由于权重的初始冗余（即专家间是完全相同的复制品），模型会经历短暂的性能停滞甚至小幅下降 。实施高效的愈合训练是确保 1.12T 模型达到预期性能的关键。

### 5.1 持续预训练（CPT）配方

愈合阶段需要投入约原预训练计算量 20% 到 40% 的资源进行持续预训练 。

1. **第一阶段：路由器热身（Router Warm-up）**：在最初的 5,000 步中，冻结所有专家权重，仅训练新增的路由参数。这一步是为了让路由器学会在 1024 个专家中进行有效选择，避免梯度直接冲击已训练好的专家 。
2. **第二阶段：对称性破缺（Symmetry Breaking）**：解冻全模型，采用带有较大扰动的随机梯度下降，促使原本相同的专家副本朝着不同的专业方向进化 。
3. **第三阶段：领域知识对齐**：使用 STEM、代码和多轮对话的混合数据流，进行领域并行训练（Domain-parallel training），这有助于 1.12T 模型在新增加的空间中填充更具专业性的知识 。

### 5.2 知识蒸馏（Knowledge Distillation）与平滑化

为了确保 1.12T 模型不丢失 560B 基座的原有能力，可以引入知识蒸馏机制。原 LFC-560B 模型作为教师模型，1.12T 模型作为学生模型，通过匹配 Logits 分布或中间层特征，强制学生模型在继承新容量的同时不背离原有的逻辑框架 。

此外，采用模型平均（Model Averaging）技术，即将愈合训练最后几个 checkpoint 进行加权融合，已被证明能有效提升模型的泛化稳定性，防止在特定数据分布上过拟合 。

### 5.3 稳定性组件：z-loss 与 PID 控制器

在扩增后的专家层中，极易出现某些专家被过度选中的现象，导致严重的负载不均衡和计算瓶颈。必须延续并强化 LFC 系列的稳定性套件：

- **隐藏 z-loss**：抑制极大的激活值，防止梯度爆炸 。
- **PID 控制器专家偏置**：动态调整路由器的偏置项，确保 1024 个专家在训练全过程中的负载偏差保持在 5% 以内 。

## 6. 硬件基础设施与部署考量

1.12T 模型对显存和互联带宽提出了极端要求。基于 80GB VRAM 的 H100 或 H800，以及 141GB VRAM 的 H200 是目前唯一的生产级选择 。

### 6.1 显存预算与并行配置

| **部署模式** | **参数精度**  | **显存需求 (1.12T)** | **推荐硬件组合**  | **来源** |
| ------------ | ------------- | -------------------- | ----------------- | -------- |
| **标准推理** | BF16          | ~2,240 GB            | 32x H100 (4 节点) |          |
| **极致压缩** | INT4          | ~630 GB              | 8x H200 (单节点)  |          |
| **训练模式** | BF16 (含状态) | ~18,000 GB           | 256x+ H800/H100   |          |



在 1.12T 规模下，单节点 8x H200 (1.1 TB VRAM) 的优势非常明显，它能够以 FP8 精度在单节点内装载整个模型，避免了跨节点 NVLink 带宽受限带来的性能损耗 。对于 H800 集群，由于 NVLink 带宽限制在 400 GB/s，必须更加依赖 ScMoE 的通信掩盖技术 。

### 6.2 DORA 系统与大规模容错

美团自研的 DORA 系统在 560B 模型训练中表现出了极高的稳定性。在 1.12T 扩增训练中，DORA 的异步流水线和弹性配置能够处理上万块加速器的协同工作 。考虑到 1.12T 模型训练周期较长，必须启用“确定性计算”（Deterministic Computation）和 SDC（静默数据损坏）检测机制，确保实验的可重复性，并在硬件故障发生时能从最近的 checkpoint 迅速恢复 。

## 7. 深入调研后的综合解决方案建议

针对 robin 的模型扩增需求，本报告提出以下三步走的最优技术方案：

### 第一阶段：架构重组与参数灌注（Phase 1: Architecture & Infusion）

1. **专家扩容**：将路由专家从 512 个扩增至 1,024 个。
2. **路由器克隆**：复制原路由器权重，并加入 $\delta = 0.01$ 的偏置噪声 。
3. **零计算扩能**：将零计算专家同步翻倍至 512 个，以抵消总参数增加带来的功耗提升 。
4. **MLA 保留**：完全保留 MLA 结构，仅需根据新的隐藏层规模进行少量参数对齐 。

### 第二阶段：结构化愈合训练（Phase 2: Structured Healing）

1. **热启动 CPT**：投入约 10T tokens 的高质量混合语料。
2. **分步解冻**：遵循“先路由器后专家”的冻结策略，维持训练早期的收敛曲线平滑 。
3. **多目标优化**：在标准交叉熵损失之外，加入与原 560B 模型的知识蒸馏损失，确保性能不发生“灾难性遗忘” 。
4. **PID 控制平衡**：严格执行动态专家偏置调整，确保 1024 个专家的利用率均衡 。

### 第三阶段：推理引擎适配与部署（Phase 3: Deployment & Optim）

1. **FP8 引擎切换**：利用 H100/H200 的 Transformer Engine，将专家层切换至 FP8 精度进行推理，实现在 1.12T 规模下维持 80+ TPS 的吞吐量 。
2. **ScMoE 参数微调**：针对 1024 专家的通信模式，微调专家并行参数，最大化计算与 All-to-All 通信的重叠率 。
3. **上下文窗口对齐**：使用 YaRN 方法对扩增后的模型进行 256k 或 1M 的长文本适配，确保 LFC 系列的 agentic 能力得到完整保留 。

## 8. 结论与性能预期

通过实施本报告提出的“专家上循环 + 结构化愈合”方案，能够在不增加推理算力需求（FLOPs per token）的前提下，将 LongCat-Flash-Chat 的总参数量成功翻倍至 1.12T 。该方案充分利用了 LFC 原有的 ScMoE 通信优化和零计算专家特性，极大地降低了模型扩容带来的硬件压力。

**预期性能目标**：

- **知识覆盖度**：在 MMLU、GPQA 等基准测试中，由于专家密度的提升，预期得分将比 560B 基座提升 2-4 个百分点 。
- **代码与 Agentic 能力**：受益于更大规模的参数空间和 N-gram 查找增强，在 SWE-Bench 和相关工具调用任务中将表现出更强的鲁棒性 。
- **稳定性**：通过 z-loss、PID 控制器和确定性计算，扩增训练过程将不会出现不可恢复的损失尖峰 。

总之，该方案为 robin 提供了一个从 560B 跃迁至万亿级参数大模型的成熟技术路线，在确保性能平滑过渡的同时，为未来的 agentic 智能演进预留了充足的参数余量。
