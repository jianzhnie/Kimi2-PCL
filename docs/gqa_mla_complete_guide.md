## 背景：注意力机制演进

### 标准多头注意力 (MHA)

在标准 Transformer 中，每个注意力头都有独立的 Query、Key 和 Value：

```
head_i = Attention(Q_i, K_i, V_i) = softmax(Q_i K_i^T / √d_k) V_i
MultiHead = Concat(head_1, ..., head_h) W_O
```

**问题**：每个头都需要缓存独立的 K/V，导致 KV Cache 内存占用巨大。

### 多查询注意力 (MQA)

所有 Query 头共享同一组 Key 和 Value：
- ✅ 内存大幅减少（KV Cache 缩小 h 倍）
- ❌ 模型表达能力下降，生成质量降低

---

## GQA - 分组查询注意力

### 核心思想

**GQA 是 MHA 和 MQA 之间的折中方案**：
- Query 头被分成 G 个组
- 每组内的 Query 头共享同一对 Key 和 Value

### 数学表达

```
输入: X ∈ R^(batch × seq × d_model)
参数:
  - num_heads = n (Query 头总数)
  - num_kv_heads = k (Key-Value 头数，k ≤ n)
  - group_size = n/k (每组包含的 Query 头数)

对于第 i 个 KV 组:
  Q_group_i = [Q_{i×g}, Q_{i×g+1}, ..., Q_{i×g+g-1}]  (g 个 Query 头)
  K_i = X W_k^i   (共享的 Key)
  V_i = X W_v^i   (共享的 Value)
  
  Attention(Q, K_i, V_i) = softmax(Q K_i^T / √d_k) V_i
```

### 特殊情况

| 配置 | 模式 |
|------|------|
| num_groups = 1 | → MQA (所有 Query 共享 1 个 KV) |
| num_groups = num_heads | → MHA (每个 Query 独立 KV) |
| 1 < num_groups < num_heads | → GQA (分组共享) |

### 示例配置

```python
num_heads = 32      # 32 个 Query 头
num_groups = 8      # 分成 8 组
→ 每组 4 个 Query 头共享 1 个 KV 头
→ KV Cache 减少为原来的 1/4
```

### GQA 的优势

1. **内存效率**：KV Cache 减少 num_heads/num_groups 倍
2. **推理加速**：减少内存带宽瓶颈，推理速度提升 30-40%
3. **质量保持**：相比 MQA，质量损失仅 1-3%（MQA 损失 5-15%）
4. **简单实现**：与标准 MHA 结构相似，易于部署

### 应用模型

- **LLaMA 2/3**
- **Mistral**
- **Qwen**
- **Gemma**
- **GPT-OSS**

## MLA - 多头潜在注意力

### 核心思想

**MLA 通过低秩压缩来减少 KV Cache，而非共享头**：
- 将 Key 和 Value 压缩到低维潜在向量 (latent vector)
- 缓存压缩后的表示，使用时再解压缩
- Query 也进行类似的低秩分解

### 数学表达

```
参数:
  - d_c: KV 压缩维度 (远小于 n_h × d_h)
  - d_c^q: Query 压缩维度

1. KV 压缩 (缓存这一步的输出):
   c_t^KV = W_DKV · RMSNorm(h_t)    ∈ R^(d_c)
   
   # 缓存 c_t^KV 而不是完整的 K, V

2. KV 解压缩 (使用时):
   k_t^C = W_UK · c_t^KV            ∈ R^(n_h × d_h)  
   v_t^C = W_UV · c_t^KV            ∈ R^(n_h × d_h)

3. Query 压缩 (不缓存):
   c_t^Q = W_DQ · RMSNorm(h_t)      ∈ R^(d_c^q)
   q_t^C = W_UQ · c_t^Q             ∈ R^(n_h × d_h)

4. 注意力计算:
   o_t = Attention(q_t^C, k_t^C, v_t^C)
```

### 压缩比例示例

以 DeepSeek-V2 为例：
```
原始 KV Cache: 2 × n_h × d_h = 2 × 128 × 128 = 32,768
压缩后: d_c = 512
压缩比: 32768 / 512 = 64×
```

### MLA 的关键创新

#### 1. 低秩投影 (LoRA-style)
```
W_full ≈ W_up × W_down
大矩阵 ≈ 上投影 × 下投影
```
- 减少参数量
- 保持表达能力

#### 2. 解耦 RoPE

MLA 将位置编码与内容分离：
```
Content: c_t^KV (压缩向量)
Position: 独立的 RoPE 编码
```
避免低秩投影混合位置和内容信息。

#### 3. 动态解压
- 缓存压缩向量（静态权重）
- 推理时实时解压（上投影是固定的）

### MLA 的优势

1. **极致压缩**：KV Cache 可减少 4-64 倍
2. **质量保持**：在某些任务上甚至优于 MHA
3. **长上下文支持**：相同内存可处理更长序列
4. **计算效率**：减少内存带宽瓶颈

### 应用模型

- **DeepSeek-V2/V3**
- **DeepSeek-R1**

## 数学公式详解

### GQA 公式

```
对于第 g 个 KV 组 (g = 1, 2, ..., n_groups):

    Query:  Q_g = {Q_{g×k}, Q_{g×k+1}, ..., Q_{(g+1)×k-1}}  (k = n_heads/n_groups)
    Key:    K_g = X · W_k^g
    Value:  V_g = X · W_v^g
    
    Attention_g = softmax(Q · K_g^T / √d_k) · V_g
    
最终输出: MultiHead = Concat(head_1, ..., head_h) · W_O

关键点: 多个 Query 共享同一对 K, V
```

### MLA 公式

```
1. KV 压缩 (缓存这一步的 c_t^KV):
   
   c_t^KV = W_DKV · RMSNorm(h_t)    ∈ ℝ^(d_c)
   
   其中: W_DKV ∈ ℝ^(d×d_c)  是下投影矩阵
         d_c << d (通常 d_c = 512, d = 4096)

2. KV 解压缩 (推理时实时计算):
   
   k_t^C = W_UK · c_t^KV            ∈ ℝ^(n_h × d_h)
   v_t^C = W_UV · c_t^KV            ∈ ℝ^(n_h × d_h)
   
   其中: W_UK, W_UV ∈ ℝ^(d_c × n_h·d_h) 是上投影矩阵

3. Query 压缩 (不缓存):
   
   c_t^Q = W_DQ · RMSNorm(h_t)      ∈ ℝ^(d_c^q)
   q_t^C = W_UQ · c_t^Q             ∈ ℝ^(n_h × d_h)

4. 带 RoPE 的注意力 (解耦位置和内容):
   
   K = K_C + K_R   (内容 + 位置)
   
   Attention = softmax(Q · K^T / √d_k) · V

关键点: 缓存压缩向量，使用时解压
```

### MLA 低秩压缩原理

```
矩阵低秩分解:

原始大矩阵 W (4096 × 16384) ≈ 67M 参数

        ┌──────────────────┐
        │                  │
        │       W          │
        │   (4096×16384)   │
        │                  │
        └──────────────────┘
                 ≈
        ┌────────┐    ┌──────────────────┐
        │        │ ×  │                  │
        │  W_up  │    │     W_down       │
        │(4096×512)│   │   (512×16384)    │
        │        │    │                  │
        └────────┘    └──────────────────┘

压缩后参数: 4096×512 + 512×16384 ≈ 2M + 8M = 10M 参数
压缩比: 67M / 10M ≈ 6.7×

原理: 大矩阵通常可以近似为两个小矩阵的乘积
```

## 内存计算对比

### 理论计算

```
假设:
  - batch_size = b
  - sequence_length = s
  - num_layers = L
  - num_heads = n_h
  - head_dim = d_h
  - compression_dim (MLA) = d_c = 512

┌───────────────────────────────────────────────────────────────────────────────┐
│  方法   │  每token KV Cache大小  │  总KV Cache (L层)      │  相对大小         │
├───────────────────────────────────────────────────────────────────────────────┤
│  MHA    │  2 × n_h × d_h         │  2 × L × n_h × d_h × s × b              │
│         │  = 2×32×128 = 8192     │  = 2×32×32×128×8192×1 ≈ 2.1 GB          │
│         │                        │                                          │
│  MQA    │  2 × 1 × d_h           │  2 × L × 1 × d_h × s × b                │
│         │  = 2×1×128 = 256       │  ≈ 67 MB                                 │
│         │                        │  (减少 n_h = 32 倍)                       │
│         │                        │                                          │
│  GQA    │  2 × n_g × d_h         │  2 × L × n_g × d_h × s × b              │
│         │  = 2×8×128 = 2048      │  ≈ 537 MB                                │
│         │                        │  (减少 n_h/n_g = 4 倍)                    │
│         │                        │                                          │
│  MLA    │  d_c (压缩向量)         │  L × d_c × s × b                        │
│         │  = 512                 │  ≈ 134 MB                                │
│         │                        │  (减少 2×n_h×d_h/d_c = 16 倍)             │
└───────────────────────────────────────────────────────────────────────────────┘

注: DeepSeek-V2 通过更激进的压缩达到 64× 压缩比
    2×128×128 / 512 = 32768 / 512 = 64
```

## 实际应用示例

### GQA 配置 (LLaMA-3)

```python
# LLaMA-3 8B 配置
num_attention_heads = 32          # Query 头数
num_key_value_heads = 8           # KV 头数 (分组数)
group_size = 32 // 8 = 4          # 每组 4 个 Query 共享 1 个 KV

# 内存节省
# 原始 MHA: 需要缓存 32 个 K 和 32 个 V = 64 个矩阵
# GQA: 只需要缓存 8 个 K 和 8 个 V = 16 个矩阵
# 节省: 64/16 = 4 倍
```

### 关键洞察

**GQA 的关键洞察**

1. **平衡的艺术**
   - MHA: 每个 Query 有独立 KV → 高内存，高质量
   - MQA: 所有 Query 共享 KV → 低内存，低质量
   - GQA: 分组共享 KV → 找到最佳平衡点

2. **为什么有效?**
   - 相邻的 Query 头往往学习相似的特征
   - 强制共享可以减少冗余，提高参数效率
   - 实验证明质量损失很小 (< 3%)

3. **配置建议**
   - 小模型 (≤7B):  4-8 组
   - 中模型 (7B-30B): 8 组
   - 大模型 (≥70B):  8 组

**MLA 的关键洞察**

1. **压缩而非共享**
   - GQA: 减少 KV 头的数量
   - MLA: 压缩每个 KV 的维度
   - 优势: 可以在更多头上保持表达能力

2. **低秩近似的威力**
   - 注意力矩阵通常是低秩的
   - 用两个小矩阵近似大矩阵: W ≈ W_up × W_down
   - DeepSeek-V2: 2×128×128 → 512 (64× 压缩)

3. **解耦 RoPE**
   - 标准低秩会混合位置和内容信息
   - MLA 将位置编码分离处理
   - 保持位置敏感性的同时实现压缩

4. **工程挑战**
   - 需要定制的 CUDA kernel 优化
   - 解压计算增加 FLOPs，但减少内存带宽瓶颈
   - 长上下文场景下收益更大

    • 选择建议:
      - 追求快速实现和部署 → GQA
      - 追求极致内存效率 → MLA
      - 平衡考虑 → GQA (当前主流选择)
    """
    print(conclusion)
    print("\n" + "=" * 80)
    print("文档生成完成")
    print("=" * 80 + "\n")


if __name__ == "__main__":
    main()
