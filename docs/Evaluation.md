# PCl-Model-1T 大语言模型评测

## 评测结果

- 当前评测结果基于训练 10000 Step 保存的Checkpoint 模型,  完成的token 数量为 4.3T。
- 推理评测框架基于 vllm + lmeval。
- 评测结果基于 MMLU Benchmark 数据集, 包含 4 个任务：人文、其他、社会科学、STEM。

### MMLU Benchmark

MMLU (Measuring Massive Multitask Language Understanding) 是一个全面的基准测试，旨在评估大型语言模型在广泛知识领域（如科学、技术、工程、数学、人文和社会科学）中的多任务能力。它涵盖了 57 个主题，从初等教育到高级专业水平，是目前衡量 LLM 综合理解和推理能力的核心评测指标之一。

### Wikitext-2 Perplexity

Wikitext-2 是一个常用于语言模型评估的数据集，包含从维基百科文章中提取的高质量文本。**Word Perplexity (困惑度)** 是衡量语言模型性能的关键指标，反映了模型对测试数据集中下一个词的预测能力。困惑度越低，表示模型对文本的预测越准确，语言建模能力越强。

##  推理评测适配过程介绍

PCL-1T-Model 项目在原始开源的 Kimi-K2-Base (`moonshotai/Kimi-K2-Base`) 架构上进行了大量改造。

### 模型适配

为了能够正确完成模型推理评测，需要完成以下关键步骤：

#### 权重转换 (Weight Conversion)

模型在训练阶段使用 Megatron-Core (MCore) 格式，而推理框架（如 Hugging Face, vLLM）通常使用 HF 格式。

- **Step1: Mcore ===> Huggingface**
    - 使用 `utils/convert_kimi2_mcore2hf.py` 进行转换。
    - **核心特性**: 支持 GQA 架构转换、QK LayerNorm 提取、128 个专家的 MoE 权重合并。
    - **并行支持**: 适配 TP (Tensor Parallel), PP (Pipeline Parallel), EP (Expert Parallel), VPP (Virtual Pipeline Parallel) 以及 DualPipe 调度算法。
    - **关键逻辑**: 需正确处理 `moe_tp_extend_ep` 标志，确保 TP 维度参与专家并行时的 rank 映射正确。

- **Step2: Huggingface ===> Mcore**
    - 使用 `utils/convert_kimi2_hf2mcore.py` 将 HF 格式权重转回分布式 MCore 检查点。
    - **并行策略**: 需指定目标集群的 TP/PP/EP 规模。
    - **混合架构**: 必须指定 `--first-k-dense-replace 2`，以确保前 2 层密集层的权重切分逻辑与训练配置对齐。
    - **加速优化**: 支持多线程读取 HF 权重及多进程并行处理 PP 阶段。

#### Hugging Face 模型适配

由于 Kimi2-PCL 采用了非标准的 **GQA + QK LayerNorm + MoE** 架构，必须在 HF 范式下重新定义模型逻辑：
- **Architecture**: 注册 `KimiK2ForCausalLM` 类。
- **Attention**: 实现带 QK LayerNorm 的 GQA。Q/K/V 的 `head_dim` 均为 128（部分版本可能涉及不对称维度适配）。
- **MoE**: 实现 Grouped Top-K 路由算法，并支持共享专家 (Shared Experts) 的并行计算。

#### vLLM 推理框架适配

为了发挥 vLLM 的 PagedAttention 和 FusedMoE 加速性能，需在 vLLM 源码级进行适配：
- **模型注册**: 在 `vllm/model_executor/models/registry.py` 中注册 `kimi_k2`。
- **算子定制**: 
    - **KimiK2Attention**: 在 RoPE 之前应用 QK LayerNorm，并确保 Attention 算子支持相应的 Head 维度。
    - **KimiK2MoE**: 集成 `FusedMoE` 算子，实现高效的跨卡路由。
- **昇腾 (Ascend) 适配**: 针对 NPU 硬件，需在 `vllm-ascend` 中确认 FlashAttention 对非标准维度的支持情况，必要时进行 Zero-Padding 处理。

#### 踩坑经验与解决方案

- **模型结构对齐**: 初期仅参考训练配置，难以直观理解 Megatron 内部复杂的 Layer 映射。
    - **方案**: 在训练代码中增加架构打印/保存逻辑，通过对比参数名和 Shape 强制对齐。
- **密集层替换逻辑**: 转换脚本若未显式指定 `first_k_dense_replace`，会导致 MoE 层与 Dense 层权重错位。
    - **方案**: 确保转换命令中的该参数与训练脚本 (`pretrain_kimi2_1t_4k.sh`) 严格一致。
- **并行 Rank 映射**: 在 `moe_tp_extend_ep` 开启时，专家权重的存储路径会发生变化。
    - **方案**: 优化 `_mp_prefix` 路径生成函数，确保在不同 TP/EP 组合下均能正确定位权重文件。

### 基于 MindSpeed-LLM 框架的推理踩坑经验

- **算子不匹配**: MindSpeed-LLM 的部分算子（如 `GroupedMatmul`）在处理 Kimi2 的 Grouped Top-K 路由时可能存在逻辑冲突。
- **解决方案**: 
    - 针对昇腾环境，优先使用 `vllm-ascend` 配合定制化的 `KimiK2ForCausalLM` 实现。
    - 在推理脚本中通过 `--trust-remote-code` 验证逻辑正确性后，再逐步迁移到高性能原生算子。

#### 踩坑经验

- 初期，仅仅参考训练的配置脚本，很难正确理解 Megatron 内部模型结构，导致模型转换过程很难对齐
- 解决方案：
    - 在训练代码增加一些保存代码，把模型的架构、参数保存起来
- 基于保存的架构、参数，正确理解 Megatron 内部模型结构，确保模型转换过程能够对齐
